## Rules & Compliance map — how CROR categories flow through Clear Board's real stations

**One brain.** Every rule category resolves to the same `ContentItem` library plus the same per-item SM-2-lite mastery model. The four stations are *delivery surfaces* over one rule set, not four apps. My slice: for each rule category, decide which surface teaches it and flag where the engine already enforces it vs. where a rule has no home.

```
CONTENT LIBRARY  public/data/{rules,signals,definitions,rule-questions}.json
  ContentItem{ domain, citation{ ref, trust }, payload }
    trust: verified | needs-review | operating-practice
    HARD GATE: needs-review may show in Reference (flagged) but NEVER in graded Drill/Exam
        |
        v
  REFERENCE  plain-first, verbatim one tap deeper, cited + trust-flagged  (every category has a home here)
        |
        v
  DRILL  graded MC, verified-only; rule items AND signal items feed ONE mastery model
        |
  THE YARD  src/stations/yard/model.js — the ONLY surface that ENFORCES rules procedurally:
        canLine / routeReady ....... CROR 104  switch lining (target reverse, intermediates normal)
        spot / secure / unsecure ... CROR 112  unattended 2+ car cut must be tied down; drop below 2 un-secures
        canKick .................... CROR 113.4/113.5  kickable-track S.I. + secured 2+ backstop
                                     + type ban (tankcar/autorack never kicked) + total<=5 / loaded<=3 + no load-onto-empties
        canPull / canSpot .......... CROR 114  lead foul (cut too long for lead) / track too full to spot clear
        (shove path) ............... CROR 115  shove only with route lined + protected
        Score = MOVES (the objective)  |  JOINTS = couplings (secondary stat)
        +  switch-list S.I.: VERIFY cars by markings — the printed list can be wrong (p2-list type)
```

**SIGNAL READER** renders aspects from our own SVG (`drawSignal`, keyed to `SignalAspect.heads`) — teaches recognition, not procedure. **RADIO** preserves exact wording in ordered steps. **EXAM** is deferred past V1 (no test-ready claim ships).

### What the engine enforces today vs. what is content-only
- **Enforced procedurally (Yard):** route/switch lining (104), securement (112), kicking limits + type bans (113.4/113.5), lead foul / track capacity (114), and the shove-needs-route gate (115 via `routeReady`). These are *active* — the move is blocked with a cited message.
- **Content-only (no procedural home yet):** yard speed / range of vision (CROR 105 — modeled as a number but never *tested* against the player), movement authority (`authority` domain, schema-reserved, empty in V1), track protection / blue-flag (`protection` domain, reserved), radio discipline (Radio station preserves wording but does not grade), and switch-list verification (the S.I. exists as a puzzle type but verification is not yet its own graded mechanic). These are the gaps.

### How a single rule travels the brain (worked example, securement / CROR 112)
1. Reference item `rule.112` (trust verified) — plain front door, verbatim one tap deeper.
2. Drill questions `q.112`, `q.112.2` (verified) feed the `securement` domain mastery.
3. The Yard *enforces* it live: `spot` auto-secures a 2+ cut (`secured[id]=true`), `unsecure`/`pull` below 2 cars flips `secured[id]=false`. Mastery rises only when retention survives ~30 days of spacing (`fam = min(1, interval/30)`), and domain mastery decays (half-life 60d) so it falls if the trainee stops — directly mitigating F2 (false-mastered).
The same item is one node touched by three surfaces — that is "one brain, not panels."

## Design suggestions

1. **Stamp every Yard rejection with the matching Reference item, not just a number** — model.js already emits cited messages (e.g. 'line its switch reverse (CROR 104)'). Make each message link to the `ContentItem.id` it enforces so a blocked move becomes a one-tap teaching moment in Reference. Closes the loop between the enforcing surface and the explaining surface — one brain, visibly. _(ties to: The Yard / CROR 104, 112, 113, 114)_
2. **Add a build-time gate that fails if any needs-review item is reachable from a graded path** — The cardinal F1 mitigation is currently a convention. Turn it into a test: walk rule-questions.json and the signal drill set, assert every graded item resolves to trust:'verified'. Fail CI otherwise. Cheap, and it makes the safety invariant un-bypassable. _(ties to: Trust gate / Drill / Exam)_
3. **Make switch-list verification its own graded micro-mechanic, reusing p2-list** — The verify-by-markings S.I. already drives a puzzle type. Surface the same skill as a Drill question form ('the list says X, the car is marked Y — what do you do?') feeding the switching domain. Builds on existing content rather than a new system, and teaches a real safety habit (trust the marks, not the paper). _(ties to: Switch-list S.I. / switching domain)_
4. **Give the kicking decision a visible, ordered checklist in the Yard, gated to verified rule text** — Kicking has five real gates (kickable track, secured 2+ backstop, allowed car type, within 5/3 limits, not load-onto-empties — all already in canKick). Show them as a checklist that lights up as conditions are met, each row citing its rule category. Turns the densest legal cluster into a felt sequence without inventing any rule. _(ties to: The Yard / CROR 113.4-113.5 + S.I.)_
5. **Differentiate the 104 route error: 'target not lined' vs 'intermediate diverts you'** — routeReady already returns two distinct messages — keep them distinct in the UI and tie each to a separate learning objective (line the target reverse / leave intermediates normal). The diversion case is the one trainees fail; naming it explicitly prevents the 'I lined a switch, why did it fail' confusion. _(ties to: The Yard / CROR 104)_
6. **Represent point protection on the shove (115) as a content-first objective before any mechanic** — routeReady gates shoving on the route, but real CROR 115 also requires protecting the leading end (visibility/a person on the point). V1: add a verified Reference + Drill item for the shoving/point-protection category [verify vs CROR/SME]; defer any sim representation. Honest coverage beats a half-modeled safety rule. _(ties to: shoving/point-protection category)_
7. **Keep yard speed / range-of-vision (105) as Reference+Drill in V1, not a fake sim check** — The Yard models speed as a constant but never tests the player against 'stop within half the range of vision.' Rather than bolt on a shaky speed mechanic in the safety path (deterministic-content law), teach 105 via verified Drill + Reference now and revisit a sim check only if it maps to a clean objective. _(ties to: yard speed / range-of-vision category)_
8. **Prevent empty schema-reserved domains (authority, protection) from reading as 0% mastery** — DomainState.mastery defaults can make an unpopulated domain look 'failed' on Home/path, implying the trainee is behind on something that does not yet exist — a false signal that feeds F3. Mark reserved domains 'not yet in this version' in the path UI instead of charting them as 0%. _(ties to: authority / protection domains, path screen)_

## Failure cases

- **F1 (cardinal): a needs-review or operating-practice item leaks into a graded Drill/Exam and is taught as settled CROR.** → Build-time assertion that every graded item is trust:'verified'; runtime guard in the Drill selector; operating-practice rendered with a visible 'practice, not a rule' flag and excluded from grading.
- **A hardcoded CROR number in a Yard message ('CROR 113.5(a)(v)') is subtly wrong, so the engine teaches a bad citation with full authority.** → Single source the rule numbers from the content library, not string literals in model.js; SME sign-off on every player-facing citation; [verify vs CROR/SME] until confirmed.
- **Trainee wins Yard puzzles by trial-and-error (spamming moves until the board is right) without internalizing the rule — competence theater.** → Score = MOVES against a solver-proven par; show par delta and which rule each rejected move violated, so 'won' with many extra moves reads as 'not yet mastered,' not success.
- **Kicking softlock/confusion: trainee tries to kick onto a track that just dropped below 2 cars (now unsecured) or onto empties, gets blocked, and reads it as a bug.** → canKick already blocks all four cases; ensure each rejection names the specific failing condition (not tied down / wrong car type / over limit / load-onto-empty) and links the rule — never a generic 'illegal move.'
- **Route (104) diversion taught as 'lining failed' instead of 'you diverted into an intermediate' — the trainee learns to randomly toggle switches.** → Keep routeReady's two distinct messages distinct in UI; the intermediate case explicitly says you would divert, tying the lesson to 'intermediates normal.'
- **Securement (112) miscounted: trainee leaves a 2-car cut thinking it is secured when a prior pull dropped it to a state where the auto-secure did not re-evaluate.** → secured[id] must re-derive on EVERY move that changes a track's count (spot raises, pull/unsecure below 2 lowers); add a unit test that walks count transitions across the 2-car boundary in both directions.
- **F3 false test-ready: Home/path implies qualification-readiness from Yard completion or activity, before retention or breadth is proven.** → No test-ready claim ships in V1 (Exam deferred); domain mastery is retention-decayed (half-life 60d) and authority/protection are explicitly 'not in this version' so the path cannot read as complete.
- **Signal recognition without the governing action: trainee can name an aspect but not what to DO on it.** → Pair every Signal Reader aspect with a Drill question on the required action tied to the indication rule; mastery for signals counts the action item, not just recognition.
- **Switch-list verification trap backfires: the puzzle's intentionally-wrong list is mistaken for a content bug and erodes trust in the app.** → Frame p2-list explicitly as 'the list can be wrong — verify by the marks (S.I.)'; on completion, surface the deliberate error so the lesson is unmistakable.

## System map

## The Yard — the switching loop as it actually exists

**Engine (byte-for-byte from switch-list, do not edit):** `src/stations/yard/{geometry,model,render,anim,sound,puzzles,solver}.js`, bridged to React by `yardSim.js` -> `Yard.tsx` via `imperativeStation`.

**The verified move pipeline (model.js):**
- Player clicks a switch target on canvas -> `lineSwitch()` toggles normal/reverse (the route-lining rule category, cited CROR 104).
- Player picks `act` (PULL/SPOT/KICK) + `count` + `track` -> `doMove()` -> `precheck()` calls `canPull`/`canSpot`/`canKick` (no mutation) -> on `ok`, `animateMove()` tweens, then commits `pull`/`spot`/`kick`.
- **Rule checks live inside the validators**, each citing a rule category: `routeReady()` = switch-lining; `LEAD_CLEAR` foul + `SPOT_CLEAR` clearance = lead-foul/clearance; `canKick` enforces kickable-track S.I., a secured 2+ backstop (securement + kicking rule categories), kickable car types (no tank/autorack/centerbeam), <=5 total / <=3 loaded, and "no load onto empties."
- **Scoring = MOVES** (`commit(cost)`): PULL=2 (back-in + pull-out), SPOT=1 (shove; pull-out deferred via the `out` flag, paid +1 on the next engine move), KICK=1. JOINTS (`state.joints`) is the secondary "also-good" stat (PULL +1, SPOT-onto +1, KICK +0). Win = `checkWin()` (set, ordered set, or DEPART with the consist on the engine). `grade()` compares to `puzzle.par`.

**Solver (solver.js):** Dijkstra over the real engine moves, cost = (moves, then joints). A Node harness (`tools/solve.mjs`, referenced but currently MISSING from the repo) is meant to prove every authored `par`/`opt` pre-ship. **Critical property: the solver calls the SAME `pull/spot/kick`, so solver and game can never disagree.** The solver deliberately does NOT model lead-staging — lead use is hand-authored, never solver-driven.

**How the loop connects to the rest of Clear Board:**
- **The Yard's only profile write is `recordWin()`** -> `profile.yard.completed[]` (a flat list of solved puzzle ids). It does NOT touch `profile.items` (the SM-2-lite `SRState`) or `profile.domains.{switching,securement}.mastery`. So today The Yard is an ISLAND — it teaches switching by doing, but contributes ZERO to the mastery/retention engine that the rest of the app calls "one brain." This is the central structural gap in my slice.
- **Drill** already serves graded MC questions tagged `domain:"switching"` (e.g. `q.104` switch-lining, `q.104.5` derails) and `domain:"securement"` (`q.112`) from `rule-questions.json`, feeding `sr.grade()` and domain mastery. So the SAME rule categories The Yard drills by doing are ALSO drilled by recall in a parallel, unconnected path.
- **Reference** holds the plain-language + verbatim text for those same rule categories (`rules.json`), trust-flagged. The Yard's rule chips (`RULES` map in puzzles.js) cite rule numbers but do NOT deep-link to those Reference items.
- **Domains schema-reserved but unpopulated:** `switching`, `securement`, `protection`, `authority`, `operating` exist in `types.ts` but their content arrays are empty — The Yard is the de-facto switching content, yet it produces no `ContentItem`s and no `SRState`.

**Layer summary for my slice:** mechanic layer (PULL/SPOT/KICK/DEPART + lining + MOVES) is mature; rule-check layer is solid for ~6 categories; the retention/mastery layer and the cross-station "one brain" wiring are essentially absent for switching; the rule-category coverage is narrow (no point-protection/shove-distance, no air/securement-test interplay, no blue-flag/track-protection).

## Design suggestions

1. **Wire The Yard into the mastery brain via per-rule-category grades** — On a win, for each entry in puzzle.rules (line/secure/kick/loads/order/verify), call the existing sr.grade() against a switching/securement SRState. Clean win at/under par grades 'correct'; an over-par or heavily-refused win grades softer (or not at all). Today recordWin() only pushes to profile.yard.completed[], so switching never decays or resurfaces — this is the single highest-value fix and it reuses sr.ts unchanged. _(ties to: switching + securement domain mastery; the 'one brain' design law)_
2. **Restore and gate tools/solve.mjs as a pre-ship CI check** — solver.js documents a Node harness that proves every par/opt, but tools/solve.mjs is absent from the repo. Re-add it and run it on every content change so a typo'd par or an unsolvable ordered puzzle can never ship. The solver already replays the REAL moves, so this is a near-free safety net for the deterministic-content law. _(ties to: deterministic, machine-verified scenarios)_
3. **Add lower-tier DEPART puzzles** — Only p6-road departs, at tier 6. Build-and-leave is a core conductor task; author a tier-1 and a tier-3 depart (unordered then ordered) so the DEPART move and manifest order are trained progressively, not introduced last. Reuses the existing goal.depart/ordered flags and the Depart button — no engine change. _(ties to: the manifest/blocking S.I. (order rule))_
4. **Deep-link Yard rule chips to Reference** — The RULES map chips (CROR 104/112/113.4...) are currently inert labels. Make each chip a tap-through to the matching Reference ContentItem (plain language first, verbatim one tap deeper, trust-flagged). This closes the loop between doing (Yard), looking up (Reference), and recall (Drill) for the same rule category. _(ties to: Reference station; plain-language-first design law)_
5. **Fill the empty tier-3 band** — Tiers jump 2 -> 4 with nothing at 3. Author tier-3 puzzles that COMBINE earlier skills (a dig-out that also needs a kick onto a verified-list backstop) so the ramp is smooth. All provable by the existing solver. _(ties to: difficulty ramp)_
6. **Make securement a player ACTION, not a free boolean** — Today state.secured is set automatically when a cut hits 2+ cars. Add a SECURE beat: leaving a 2+ car cut unattended (engine departs the track) should REQUIRE the player to secure it, with a refusal if they don't — turning the securement rule from a passive check into a trained decision. Requires an engine edit, so gate it behind a full solver re-run and a parity test. _(ties to: the securement rule category [verify hand-brake count vs CROR/SME])_
7. **Add a point-protection decision to shoving** — The shove rule (CROR 115) is cited but the player never makes a protection choice. Add an optional 'protect the point' confirm before a SPOT into an occupied or blind track; teach that you shove only with the route lined AND protected. Keep it a one-tap acknowledgment in V1 so it doesn't inflate MOVES. _(ties to: the shoving/point-protection rule category [verify vs CROR/SME])_
8. **Narrate yard state for non-visual play** — All feedback except the Moves/par/Joints readout and the banner is canvas-only (switch positions, car marks, track contents). Add an accessible, screen-reader text mirror of the yard state and switch lining, plus Web Speech read-aloud of the work order and refusal messages, so a blind trainee can complete a puzzle. Accessibility is testable acceptance criteria here, not polish. _(ties to: accessibility design law)_

## Failure cases

- **The Yard teaches switching by doing but never feeds mastery, so a trainee who aced 19 puzzles months ago shows no decay and could be falsely surfaced as switching-mastered (F2).** → Wire per-rule-category sr.grade() on win (suggestion 1); let switching mastery decay by the same 60-day half-life as every other domain so it reflects retention, not a one-time completion.
- **A puzzle ships with a wrong par because tools/solve.mjs is missing and no CI re-proves it; the app then teaches that an inefficient line is 'par' or marks a true optimum as over-par.** → Restore the solver harness and gate content on it; the solver replays the real engine moves so a mismatch is caught deterministically before ship.
- **A player stashes a blocker onto the only viable scratch track and softlocks with no winning continuation, but the app gives no signal — they grind moves thinking a solution remains.** → Run the solver from reachable states (or pre-prove every puzzle is softlock-free), and always keep Reset one tap; optionally detect 'no solution remains' and offer a non-judgmental reset prompt.
- **Player learns the unsafe habit of kicking because it scores best, then over-generalizes that kicking is always allowed.** → The engine already refuses kicks off kickable tracks / non-secured backstops / banned car types / over-limit / load-onto-empties with rule-cited messages — preserve those, and ADD a Reference deep-link so the player sees WHY (the S.I. is yard-specific), reinforcing 'you must know YOUR yard's special instruction.'
- **Player ignores the switch-list verify step or certifies wrongly (p2-list), internalizing 'trust the printed list' — the exact opposite of the safety lesson.** → Keep the hard gate (no moves until certified) and the specific corrective feedback ('AS75 has no CN 318044 — that car isn't where the list says'); never let a wrong certification pass silently.
- **Adding a new player action (SECURE / protect-the-shove) silently changes the cost model and desyncs solver from game, so pars become wrong without anyone noticing.** → Treat any engine edit as a parity event: re-run the solver across all puzzles and assert game-replay == solver-par for each before merge.
- **Refusal messages cite specific CROR numbers (113.4, 113.5(a)(v), 114) that may not exactly match the current CROR; a wrong citation in a safety trainer is the cardinal sin (F1).** → Audit every citation in puzzles.js RULES and model.js refusals against CROR/SME; downgrade any unverified one to a rule-category phrasing with a [verify vs CROR/SME] flag and keep it out of graded refusals until verified.
- **Canvas-only feedback means a screen-reader or keyboard-only user cannot line a switch (clicks have no keyboard equivalent) and is fully locked out of the station.** → Add keyboard switch-lining and a text/ARIA mirror of yard state; make this a release-blocking accessibility acceptance test, not a nice-to-have.
- **Over-par wins still show a celebratory win banner, teaching that an inefficient line is fine; the trainee never learns to plan moves ahead.** → Differentiate feedback: 'Done in N moves — par is P, try Watch optimal to see the 2-move line' for over-par, reserve the green win tone for at/under par; tie the graded mastery bump to at/under par.

## Validation steps

- Run the restored tools/solve.mjs across all puzzles; assert every authored par equals the solver's par and that replay(puzzle, opt).moves == par and .ok == true for each. Zero mismatches required.
- Parity test: for each puzzle, drive the game's pull/spot/kick directly through the opt line and assert the resulting state.moves and checkWin() match the solver. Any future engine edit must keep this green.
- Softlock check: from a sample of reachable mid-game states per puzzle, confirm a winning continuation still exists (or that the puzzle is provably softlock-free); confirm Reset always restores a solvable fresh state.
- One-brain check: complete a kick puzzle and assert the relevant switching/securement SRState's seen/correct incremented and domain mastery moved — proving The Yard now feeds the brain, not just completed[].
- Citation audit: enumerate every rule number in puzzles.js RULES and model.js refusal strings; each must be verified against CROR/SME or carry a [verify] flag and be excluded from graded refusals. No needs-review citation in a graded path.
- Accessibility gate: keyboard-only user can line switches, pick and execute a move, and win a puzzle; screen reader announces yard state, switch lining, Moves/par/Joints, and refusal reasons; all controls >=44px; reduced-motion makes animation near-instant (setSpeed(6)).
- Audio gate: with the audio setting on, the work order and refusal banners are read aloud via Web Speech and degrade gracefully offline.
- Scoring-edge test: a finishing SPOT costs 1 and a mid-job SPOT costs an effective 2 (deferred pull-out) identically in game and solver; confirm pull,pull,spot == 5 and a kick saves the deferred pull-out on a representative puzzle.
- Manifest/ordered test: p1-sort (par 19) and any new ordered/depart puzzle are solvable within solver caps and the game accepts ONLY the exact manifest order for the win.
- Long-car test: p5-long's lead cap ('about seven') matches LEAD_CLEAR/CARLEN exactly so the two-cut par-7 line is the proven optimum and a one-grab attempt is correctly refused as a lead foul.

# Agent C — Training System (drill/recall/mastery/progression layer)

## System map

## Agent C slice — the training layer, as it actually exists in clear-board

**The one mastery model (real code path).** Every graded answer flows: station → `recordAnswer(item, correct)` in `AppContext.tsx` → `grade(prev, correct)` in `core/sr.ts` (SM-2-lite) → writes `profile.items[item.id]` (`fam`, `ease`, `interval`, `due`, `seen`, `correct`, `lastSeen`) → `recomputeMastery(profile, content)` in `core/store.ts` → `profile.domains[d].mastery` = mean `decayedFam` of that domain's **drillable** (verified + answerable) items → Home's `stageFor()` rolls live domains into one stage label + a single % bar. That is the spine. It is correct and it works.

**The hole in the spine — only one station writes to it.** Today **only `Drill.tsx`** calls `recordAnswer`. The map of who-writes-what:
- **Drill** → `recordAnswer` → SR + mastery. ✅ The whole brain.
- **The Yard** (`yardSim.js`) → only `ctx.profile.yard.completed.push(id)` (a binary list) + its OWN local `grade(state, puzzle)` for **Moves/par** scoring. It never touches `profile.items` or the `switching` domain. So `switching` mastery is structurally stuck at 0 — the hardest, most job-real station contributes nothing to the path.
- **Radio** (`radioSim.js`) → only `profile.radio.done = true` (one boolean for the whole domain).
- **Signal Reader** (`Signal.tsx`) → renders SVG; signals are only *graded* when they appear inside Drill (`buildQ` kind `'signal'`). The standalone reader is look-only.

So "one brain, not panels" is true for **content** (one library, one `recordAnswer`) but only half-true for **practice**: three of five stations are panels that don't feed retention. **This is the central thing my slice must fix, and it's an extension of an existing function, not a new system.**

**Selection / scheduling (real code).** `Drill.tsx` builds a session once per `(content, domain, sessionKey)`: pool = `drillable` items for the domain (or all live domains), filter due via `isDue` (`due <= now`), `chosen = shuffle(due.length ? due : all).slice(0,10)`. Distractors come from same-`type` siblings. **Tier is loaded but ignored** — `rule-questions.json` carries `tier` 1/2/3 (55/104/67) yet `buildQ` picks a random question for the rule and selection never orders or gates by tier. So a first-day trainee and a near-qualified user see the same flat random-10. The ramp the SPEC promises ("gentle ramp", D-0002 path) is **specified but not implemented** at the question level.

**Progression / gating (real code).** `profile.path = { stage, unlocked:['reference','drill'], testReady:false }` exists but **nothing reads `unlocked`** — Home shows every station always. `stageFor()` is a pure label off mean mastery; it unlocks nothing. `domains[d].lastDrill` is written, never read. `testReady` is hard-`false` by design (D-0013, Exam deferred). So "the path that ramps gently" is currently a **progress thermometer, not a path** — no breadth gate, no retention gate, no sequencing.

**Trust safety (works — keep).** `gradable`/`drillable` filter `trust==='verified'`; the 4 `needs-review` signals (410/418/433A/440 per D-0009) are correctly barred from Drill and only `drillable` rules (those with an authored question) are eligible. F1 mitigation is real and must survive any change I propose: **anything I add that grades MUST route through `drillable`/`gradable`, never raw `byDomain`.**

**My slice's job, concretely:** (1) make The Yard, Radio, and the Signal reader write to the ONE mastery model as scenario/skill items, so `switching`/`radio` domains stop reading 0; (2) turn the flat random-10 into a tier-aware, due-first, breadth-respecting selector that ramps; (3) turn `path` from a label into a real (but gentle, never-locking) breadth+retention gate that points the next step; (4) specify — not ship — the deferred test-ready/Exam retention bar so Jordan+SME can set the number later.

# Agent D — Error Checking & Feedback / Correction System

## System map

## Agent D slice: where errors are born, caught, explained, and remediated

Clear Board has **one mastery brain** fed by several error sources. My slice turns every wrong action into (a) honest in-the-moment feedback and (b) a future drill — without ever teaching something false (F1).

**Error sources → one feedback contract → one mastery model**

1. **The Yard (`src/stations/yard/model.js`)** — the only station with a *physics-true* validator. Moves are PULL / SPOT / KICK / DEPART. Validation already returns `{ ok:false, msg }` carrying a rule citation (confirmed in code):
   - `canPull` → route check (CROR 104 lining) + lead-foul (CROR 114, `cutLen > LEAD_CLEAR`).
   - `canSpot` → route check + track-full / foul-point (CROR 114).
   - `canKick` → kickable-track S.I., secured 2+ backstop (113.4/113.5), non-kickable type (tank/autorack), `total>5`, `loaded>3`, and the loaded-onto-empty ban.
   - `routeReady(state,id)` → target switch REVERSE + every intermediate NORMAL (CROR 104) — the mis-lining trap.
   - Securement (CROR 112) is implicit: `pull` dropping a track below 2 sets `secured=false`; `spot`/`kick` of 2+ sets `secured=true`.
   These refusals are the richest error surface in the app and the template for the rest.

2. **Drill (`rule-questions.json`)** — graded MCQ over rules+signals; each item has `explain` + `trust`. A wrong answer is the SM-2-lite *miss* event (ease −0.2, interval reset, re-show same session).

3. **Signal Reader (`signals.json` + `src/core/signal.ts`)** — naming/aspect confusion (e.g. 405 CLEAR vs 406 CLEAR-TO-LIMITED). Drillable as graded recall.

4. **Reference** — *not* an error surface (no grading) but the **destination of every correction link**: plain language first, verbatim CROR one tap deeper, trust-flagged.

**The unifying contract (my core recommendation).** Today only the Yard emits rich refusals. Standardize a single `MistakeEvent { itemId, domain, ruleCategory, errorClass, severity, correctAction, citationRef, trust, station, ts }` emitted by *every* station. It does three things, always in this order: (1) **stop the unsafe action** (Yard refuses the move; Drill marks wrong); (2) **explain** with cause → rule category → correct action, linking into Reference; (3) **feed the brain** — a miss decays `fam`, and high-severity safety errors additionally tag the item for **mistake-driven remediation** (a priority re-queue ahead of normal SM-2 due dates).

**Severity ladder (drives feedback tone + remediation weight):**
- **S3 safety-critical** — securement skipped, route fouled/mis-lined, kick a tank/autorack, kick loaded-onto-empty, depart with an unsecured cut behind. These get the strongest correction and force a remediation drill before the domain mastery bar can rise.
- **S2 rule violation** — over-kick-limit, lead foul, spot a too-full track, wrong switch lined.
- **S1 inefficiency** — legal but wasteful (spot where a kick was allowed, extra pulls, unnecessary joints). Coached, never punished as "wrong."

**How scoring connects.** MOVES is the objective; JOINTS a secondary stat. Inefficiency (S1) is feedback against *par* (the solver-verified par in `tools/solve.mjs`), never a safety failure — so the app must visibly separate "unsafe" (red, blocking) from "inefficient" (amber, advisory). Mastery (`fam`, half-life-60d decay) must only move on **verified** items; needs-review content can teach in Reference but can never grade (F1 firewall).

## Comparisons

### How the Yard delivers correction feedback after an unsafe/invalid move

| Approach | Strengths | Weaknesses | Best use |
|---|---|---|---|
| Block-and-explain (preventive refusal) | Matches the existing engine — model.js already returns {ok:false, msg} with a CROR citation; unsafe action never happens, so no false-success is ever modeled; safest for an S3 securement/foul/kick error; zero softlock risk. | Learner never SEES the consequence of the bad move, so the 'why it's dangerous' can feel abstract; risk of trial-and-error 'poke until it lets me' if the message is thin. | Default for ALL S3 safety-critical errors (securement, foul, mis-line, kick tank/load-into-empty) — the safety path must never let the bad move resolve. |
| Allow-and-show-consequence (sandbox replay) | Powerful learning — show the cut roll past the foul point, or the unsecured cut roll out; consequence is visceral and memorable; great for the 'learn by doing' audience. | Risk of teaching the wrong muscle memory; requires a clean 'this was unsafe, here's what happened, now undo' framing or it models success; heavier to build (animation of failure states). | Optional, opt-in 'show me why' on an S2 inefficiency or a NON-safety mistake — never the live path for an S3. |
| Post-hoc coach (let the puzzle finish, grade at the end) | Doesn't interrupt flow; lets the learner build a whole plan then reviews it against par + safety; mirrors a real debrief. | Terrible for safety errors — a fouled route or skipped securement must be caught AT the moment, not 10 moves later; lets unsafe states persist and compound; weakest F1/F2 protection. | Only for S1 efficiency debrief at puzzle WIN ('par was 5, you used 7, here's the kick you missed') — layered on top of, never replacing, live S3 blocking. |

### How a mistake feeds future drills (remediation engine)

| Approach | Strengths | Weaknesses | Best use |
|---|---|---|---|
| Pure SM-2-lite (existing) — miss just resets interval | Already specified and simple; one model; ease/interval math is honest about retention; no new system. | Treats a trivial slip and a safety-critical foul identically; a securement miss waits its turn in the queue instead of being hammered until solid. | S1 and routine S2 misses where normal spacing is enough. |
| Severity-weighted priority re-queue (recommended) | Builds directly on SM-2-lite — an S3 miss jumps the queue and re-shows same session until one clean recall, so the cardinal safety rules are over-practiced; still one brain, just a priority lane. | Must guard against frustration loops (same hard item forever); needs a cap and a 'cool-down' so it doesn't become punishing. | The default for all safety-critical (S3) misses across Yard and Drill — the anti-false-confidence core. |
| Adaptive distractor injection (targeted confusion drills) | When a learner confuses CLEAR vs CLEAR-TO-LIMITED, the next drill deliberately pairs those two as choices — attacks the exact confusion, not the topic in general; very efficient. | Needs a curated confusion-map (which items are mutually confusable) and verified distractors; more content work; can feel 'gotcha' if overused. | Signal Reader aspect pairs and rule near-misses where the SAME wrong answer keeps recurring. |

| Approach | Strengths | Weaknesses | Best use |
|---|---|---|---|
| Two-channel (red=unsafe/blocking, amber=inefficient/advisory) | Keeps the safety line bright and unmissable; an S1 efficiency note never looks like a safety failure (protects confidence of an anxious learner); maps cleanly to MOVES-vs-safety split. | Requires discipline tagging every message; two visual languages to maintain. | The standard for the whole app — never let inefficiency feedback read as 'you did something dangerous.' |

## Agent E slice: where "difficulty & fun" actually lives in Clear Board

Clear Board has TWO independent progress/scoring engines plus the path that frames them. My slice spans both and how they feed the one brain.

**1. The Yard (`yardSim.js` + `yard/puzzles.js` + `yard/model.js`)** — the only station with explicit difficulty.

- Yard ⇄ mastery: today the Yard writes only `yard.completed`. It does NOT feed the SR/mastery model for the `switching`/`securement` domains (schema-reserved). My biggest structural lever is making a *clean win that beats par* on a kick/securement/order puzzle nudge the matching rule-category's `fam`, so Yard skill and Reference/Drill knowledge converge into one number.
- Path ⇄ both: Home should turn "completed + due + beaten-par" into one calm "do this next," not two scoreboards. Difficulty is delivered by *what the path serves next*, not by a hard mode the anxious learner has to choose.

So my job: add **just enough** progress feel, pacing, recovery, and ramp to the Yard and Drill — built on `par`, `tier`, `completed`, and the SR state that already exist — without inventing a second economy, without a clock that punishes the anxious, and without ever letting "fun" reward an unsafe move the engine should refuse.

### How the pieces connect (the shared brain)
store.ts: CONTENT_FILES (definitions+signals+rules) and QUESTION_FILES (rule-questions) load into one library; questionsByRule indexes MC by ruleId. drillable() = verified AND answerable (a rule needs >=1 authored question). Drill.tsx builds one 10-item session from due items across all live DOMAINS, calls buildQ per item-type, and recordAnswer updates per-item SM-2 + domain mastery. Reference.tsx renders EVERY item (including the same signals/rules/definitions) as a lookup. So definitions/signals/rules each appear in BOTH Reference (look up) and Drill (test) by design — that's healthy reuse, not redundancy. The redundancy is when two *graded* paths score the same objective into the same domain, or when a sim teaches a rule the quiz already grades but the sim's result is thrown away.

### Cross-domain mislabels that read as redundancy

- rule.112 (securement) carries 17 questions — the single most-tested rule — but the `securement` domain has only 3 rule items. Inspected: the 17 are genuinely well-differentiated facets (unattended definition, effectiveness test, grade/tonnage, loco exception, air-brakes-alone), NOT near-duplicates. ## Failure cases

- **A learner maxes the signals domain entirely through text MC (signals rule-questions) and never once reads an SVG aspect, then can't identify a real signal on the ground — classic F2 false-mastery on a safety-critical skill.** → Require >=1 correct SVG 'What signal is this?' recall before signals counts as mastered; link 1:1 signal rule-questions to their item so the recognition skill is actually exercised.
- **A learner clears all 19 Yard puzzles but switching mastery stays low (sim doesn't feed mastery), reads it as 'I'm bad at switching', and disengages — a false-LOW mastery that demoralizes exactly the anxious audience Clear Board targets.** → Surface Yard completion on the path immediately (V1); decide the mastery-contribution question post-V1, but never let the bar contradict demonstrated competence silently.
- **Two near-identical quiz stems (inherited from CN Conductor Trainer and training-assistant banks) on the same ruleId appear in one 10-question session, making the drill feel padded and teaching nothing new the second time.** → Content-lint for >80% stem overlap on the same ruleId+answer; cap per-ruleId picks per session in Drill's builder so even legit facets spread out.
- **rule.112 dominates every securement session (17/20 questions), so securement mastery measures '112 recall' not securement breadth, hiding gaps in the other securement rules.** → Per-ruleId cap in session selection; ensure recomputeMastery averages per ITEM so one over-tested rule doesn't inflate the domain.
- **rule.12 hand signals sits in the radio domain, so a 'radio' drill mixes voice procedure with arm signals; the learner conflates two communication modes and gives a hand signal where a radio call is required (or vice versa) — a real operating-confusion trap.** → Re-file hand signals to the correct domain (verify vs CROR/SME); keep radio drills to voice procedure only.
- **Adding a new signal in future, an author adds it as BOTH a signal item AND a signals rule-question (because both patterns exist), silently re-introducing the double-grade after it's been fixed.** → Build-failing content-lint on CROR-ref collisions + the written 'where does content live' authoring rule baked into the schema doc.
- **The p2-list 'trust the marks not the list' lesson is also explained in Reference and possibly hinted in a radio/switch-list-verify step, so the same Special-Instruction is delivered three times with no escalation — wasted attention and a sense of repetition that erodes trust in the app's editing.** → Designate the Yard puzzle as the canonical place this S.I. is *practiced*; Reference only *defines* it; remove any third redundant mention.
- **Merging the radio walkthrough into the radio quiz to 'dedupe' accidentally turns a gentle confidence-builder into a graded test, alienating the rulebook-failed audience.** → Differentiate-and-keep, do NOT merge: walkthrough = wording fluency (ungraded practice), quiz = graded knowledge; label both so neither reads as redundant.

## Validation surface of Clear Board (what I gate)

Clear Board has **one brain** — `core/store.ts` builds a single `Content` graph (items + questions indexed `byId` / `byDomain` / `questionsByRule`), and `core/sr.ts` drives one retention model. My QA loop wraps every path that turns content into a graded judgment.

**Content → trust gate → graded path (the F1 firewall).** `store.ts` exposes the load-bearing safety boundary: `gradable(items)` keeps only `trust === 'verified'`; `drillable(items, content)` keeps verified AND answerable (a `rule` needs ≥1 authored question in `questionsByRule[id]`). Confirmed live in the data: 4 signals are `needs-review` and are excluded from Drill; all 100 rules + 226 questions are `verified`. **Reference** shows needs-review (flagged); **Drill/Exam** never can. My rule-mapping and quiz-validity gates sit on top of this filter — the gate is only as safe as the data feeding it, so the harness must RE-ASSERT the invariant at build time, not trust the runtime filter alone.

**Drill (`stations/Drill.tsx`) — one loop, three question shapes.** `buildQ` produces `rule` (random authored Q from `questionsByRule`, shuffled `choices`), `signal` (renders an SVG aspect from `payload.aspects`; distractors = other signal titles), and `text` (definition term↔plain, both directions; distractors via `distractors()` over same-type pool). Session = due items (`isDue`) else all, sliced to 10; refuses if `<4` drillable. Every answer → `recordAnswer(item, correct)` → `sr.grade` → domain mastery. My drill-effectiveness + quiz-validity gates check: does the question test the target micro-skill, are distractors plausible-but-wrong (not absurd, not a paraphrase of the answer), and for `signal` does the rendered aspect actually match the claimed title.

**The Yard (`stations/yard/`) — the deterministic engine.** `model.js` is the rules brain: `pull/spot/kick` with the MOVES cost model (PULL=back-in+pull-out=2; SPOT=shove-in=1 with a deferred +1 pull-out paid on the next engine move; KICK=1, 0 joints), `routeReady` (CROR 104: target reverse + intermediates normal), securement (an unattended 2+ cut auto-secures; drops below 2 → un-secures), and `canKick` enforcing kickable-track S.I., 2+ secured backstop, type bans, ≤5 total / ≤3 loaded, and no-loaded-onto-empties. `checkWin` accepts set, exact `ordered`, or `depart` (consist on the engine). `solver.js` Dijkstra's over states cost=(moves, then joints) using the SAME engine moves — so solver and game CANNOT disagree — and exposes `solve()` (par + optimal line) and `replay(puzzle, line)` (does a line win, in how many moves). `puzzles.js` holds **19** fixed puzzles (briefing says ~22), each with a hardcoded `par`, tiers 0/1/2/4/5/6 (no tier 3).

**Where the gate is missing.** The briefing states `tools/solve.mjs` "proves every par before ship." It does **not exist** — there is no `tools/` dir, no test script in `package.json` (`lint` = oxlint only, no typecheck, no test runner), and `.github/workflows/deploy.yml` is deploy-only. The solver is import-ready but only runs in the browser render path. **So today nothing machine-verifies that the 19 hardcoded `par` values equal the solver's true optimum, or that each puzzle is even solvable.** That single committed harness is my highest-priority deliverable — it converts the engine's existing self-consistency into a real pre-ship gate.

**How it all connects to scoring.** Yard `grade()` compares `state.moves` to `puzzle.par` → "Par!" vs over. SR `grade()` maps a Drill answer to interval/ease/fam → domain mastery (decayed, half-life 60d) → `path.testReady`. My loop validates each arrow: mechanic→rule-category (rule-mapping), puzzle par→solver (scenario-correctness), question→micro-skill (drill-effectiveness), distractor→understanding (quiz-validity), and the conservative bar that must hold before `testReady` could ever flip true.

- Verify the signal Drill correctness loop end to end: `buildQ` for a `signal` picks a random aspect from `payload.aspects` and asserts `correct = item.title`; confirm in `signals.json` that every aspect under a title is genuinely diagnostic of THAT title (no aspect that is ambiguous between two signal names) — an unverified mapping here is an F1 hit even on 'verified' data.
- Define the playtest gate for The Yard: each tier-representative puzzle must be (a) winnable at par by a tester following the in-game hints, (b) not softlockable into an unrecoverable state, (c) clear about WHY a refused move was refused (the `refuse()` msg must name the rule category).
- Audit `radioSim.js` (radio-script practice, 'exact wording preserved') for an effectiveness gate even though Radio is schema-reserved — ordered-step matching must reject paraphrase, and the source script must be cited+trust-flagged before it can grade.
- Define the CONSERVATIVE bar before any 'test-ready' claim can ship (Exam is deferred): require an SME-set breadth threshold (every live domain's verified items above a fam floor), a retention threshold (mastery held across ≥1 spacing interval, not same-session), zero needs-review in the exam pool, and explicit SME sign-off recorded in DECISIONS — until then `path.testReady` must be hard-pinned false.
- Add an accessibility acceptance harness (a11y is testable criteria, not polish): assert every interactive control ≥44px, `prefers-reduced-motion` kills Yard animation, the read-aloud (`speak`) fires on each new Drill prompt and degrades silently offline, focus moves to the feedback/Next control after answering (`autoFocus` is present — verify it lands), and feedback uses `aria-live` (confirmed `aria-live=polite` on the feedback block).
- Define a determinism check: the safety path must never depend on fragile runtime generation — confirm the solver is used only to VERIFY hand-authored puzzles, never to generate a graded puzzle at runtime (the solver comment claims this; assert no puzzle is procedurally generated in a graded flow).
- Define the revision loop: when a harness fails, the fix order is (1) is the DATA wrong (re-flag to needs-review, pull from graded path), (2) is the PAR wrong (re-author or accept solver's number), (3) is the ENGINE wrong (fix model.js + re-run all 19). Never 'fix' a par by editing the solver to agree.
- Edge cases to cover in the harness: a puzzle whose par requires a KICK (tier 4 loads/empties) — assert the optimal line actually uses kick and saves the deferred pull-out; a DEPART puzzle — assert win is on the engine in `ordered` sequence; a 'bad switch list' puzzle (tier 2 trust-the-marks) — assert the authored solution accounts for the marking discrepancy, not the printed list.

## Comparisons

### How to gate Yard puzzle correctness (par + solvability) in a no-CI, offline app

| Approach | Strengths | Weaknesses | Best use |
|---|---|---|---|
| Committed Node harness `tools/solve.mjs` over the shared engine (the briefing's intended design) | Reuses the EXACT `model.js` moves the game uses, so solver and game cannot disagree by construction; deterministic; zero new deps; runnable locally as `npm run check` and as a pre-deploy step; proves par == optimum AND replays the line to a win. Catches a wrong hardcoded par, an unsolvable puzzle, and an engine regression in one pass. | Dijkstra is bounded by `maxStates` (150k) / `maxMoves` (28) — a pathological puzzle could return null on a legit-but-deep solution, masking a real par as 'unsolvable'. Must tune caps per tier and treat null as a hard failure to investigate, not a pass. | The default and required gate for all 19 fixed puzzles; run on every commit that touches `puzzles.js`, `model.js`, `geometry.js`, or `solver.js`. |
| Golden-line fixtures (store each puzzle's known-good optimal move list, replay-only) | Extremely fast and stable (no search); pins the EXACT intended teaching line, not just a par number; immune to solver cap tuning; doubles as documentation of 'how this puzzle is meant to be solved.' | Does NOT prove the line is OPTIMAL — a shorter line could exist and the recorded par would be wrong without anyone noticing; fixtures must be regenerated when a puzzle changes, creating maintenance drift. | As a COMPLEMENT to the solver harness: solver proves optimality, fixtures pin the intended pedagogy and catch silent engine changes cheaply between full solves. |
| Manual playtest sign-off only (a human solves each puzzle to par before ship) | Catches 'is this actually fun / teachable' which no solver can; surfaces unclear refusals and softlocks a search won't feel; required anyway for the playtest gate. | Not repeatable, not a regression guard — an engine change after sign-off silently breaks pars; humans miss that a 7-move par is actually beatable in 6; does not scale to 19 puzzles on every change. | The FINAL gate layered on top of the automated harnesses, never a substitute for them; one tester pass per release, focused on clarity/fun, trusting the math to the harness. |

### How to validate quiz/drill VALIDITY (understanding vs memorization)

| Approach | Strengths | Weaknesses | Best use |
|---|---|---|---|
| Distractor-quality static analysis (automated, over the question bank) | Scales to all 226 questions; catches the answer appearing verbatim in the stem, a distractor equal to the answer, distractors that are themselves true, and implausibly silly distractors (length/format outliers); runs in `content-check.mjs` with no deps. | Cannot judge semantic plausibility — two distractors can be statically distinct yet test recall not understanding; needs an SME eye for the subtle cases. | First-pass filter on every question add/edit; flags candidates for SME review rather than auto-approving. |
| Micro-skill tagging + coverage matrix (each question tagged to one learning objective) | Makes 'does the drill reinforce the EXACT target micro-skill' checkable — a coverage report shows which objectives are over/under-tested and whether a 'verified' rule has any question at all (drillable already requires ≥1, this measures depth); reuses the existing `ruleId`/`domain` fields. | Requires authoring a tag per question (one-time cost on 226); tags can rot if not maintained with the bank. | For breadth/retention reporting and as a precondition of the 'test-ready' bar — you cannot claim exam-ready coverage without the matrix. |
| Item-response sanity from real usage (flag questions everyone gets right or everyone gets wrong) | Surfaces a broken question empirically — 100% correct = too easy / answer obvious in stem; 0% correct = answer likely wrong or stem ambiguous; uses data already in the SR profile (`seen`/`correct`). | Needs real users and is local-only/no-server by design, so signal is sparse and never aggregated; cannot gate a fresh ship; privacy-by-design means you can only see ONE device's stats. | A self-check a power-user/SME runs on their own device during deep playtest; never the primary gate, but a cheap smoke signal for an obviously broken item. |

### Where to enforce the trust/safety (F1) firewall

| Approach | Strengths | Weaknesses | Best use |
|---|---|---|---|
| Runtime filter only (current state: `gradable`/`drillable` in store.ts) | Already implemented and correct; centralizes the rule in one place; impossible to drill a needs-review item even if data slips through. | Silent — a mis-flagged item just vanishes from drills with no alarm; a typo in `trust` (e.g. 'verfied') would be treated as non-verified and silently drop GOOD content, OR a stray valid-but-wrong item flagged 'verified' sails straight into grading. The runtime can't tell you the DATA is wrong, only obey it. | Keep as the last line of defense; never the only line. |
| Build-time content assertion (committed `content-check.mjs`, fails the deploy) | Turns the invariant into a hard pre-ship gate; catches `trust` typos, an `answer` not in `choices`, a question pointing at a missing rule, duplicate ids; runs offline with no deps; blocks the Pages publish on red. | Cannot judge whether a 'verified' item is ACTUALLY correct — only that the wiring is sound; correctness of the claim still needs SME. | The required complement to the runtime filter — wiring soundness automated, claim-truth to SME. |
| SME sign-off ledger (DECISIONS-recorded human verification per item batch) | The only thing that can certify a rule/signal is TRUE, not just well-formed; creates an auditable trail (matches the project's 'docs are co-equal' law); required before any item flips needs-review→verified. | Slow, human, doesn't scale automatically; risk of rubber-stamping under time pressure. | The gate for trust-state TRANSITIONS and for the conservative test-ready bar; automate the wiring, reserve humans for truth. |

## Design suggestions

1. **Ship the missing `tools/solve.mjs` harness this slice — it is the gate the docs already promise** — Create the Node harness that imports `model/solver/geometry/puzzles.js` and asserts, for all 19 puzzles: solvable, `solve().par === puzzle.par`, and `replay(solve().opt)` wins. Exit non-zero on any failure. The solver already reuses the real engine moves, so this is a thin wrapper — but until it is committed and run, every par is an unverified claim and the briefing's 'solver proves every par before ship' is false. _(ties to: The Yard / scoring (par integrity))_
2. **Add `npm run check` and block the Pages deploy on it** — Compose solve.mjs + content-check.mjs + `tsc -b --noEmit` + oxlint into one `check` script, and add it as a required step before the build in `.github/workflows/deploy.yml`. Today `lint` is oxlint-only and CI just deploys — a broken par or a needs-review item leaking into a drill would ship silently. One red gate before publish is the whole no-CI strategy. _(ties to: All stations (pre-ship gate))_
3. **Promote the runtime F1 filter into a build-time assertion** — `gradable`/`drillable` correctly exclude needs-review at runtime, but a `trust` typo or an `answer` not in its `choices` is invisible. Add `content-check.mjs` to fail the build on: any non-{verified,needs-review,operating-practice} trust value, any answer not in choices, any drillable rule with zero questions, any question.ruleId missing from rules.json, any duplicate id. This makes the cardinal-sin firewall auditable, not just obeyed. _(ties to: Reference / Drill (F1 mitigation))_
4. **Fix the SR spec drift and pin it with a golden-vector test** — `sr.ts` uses `ease += 0.05` with a 2.8 ceiling; the briefing/SPEC say `+0.1` with no ceiling. Pick canonical, correct the loser, and add a tiny pure-function test that replays a fixed answer sequence and asserts exact interval/ease/fam/due — so the anti-false-confidence engine (F2) can never silently drift again. Note the decay (half-life 60d) so mastery FALLS when idle, as intended. _(ties to: Mastery/retention (F2))_
5. **Add a 'distractor collision' check to content-check.mjs** — `distractors()` draws random same-type items, which can surface a distractor that is itself a TRUE answer to a definition stem, or a near-identical signal title. Statically flag: distractor == answer, distractor text is a substring/superset of the answer, or (for signals) two titles sharing the same aspect encoding. Flagged items go to SME review. This protects 'tests understanding not memorization' from accidental ambiguity. _(ties to: Drill (quiz validity))_
6. **Layer golden-line fixtures under the solver for cheap regression catches** — Record each puzzle's intended optimal move list as a committed fixture; replay-only on every change (fast, no search). The solver proves optimality; the fixture pins the intended TEACHING line and instantly catches a silent engine change between full solves. Together they cover both 'is the par right' and 'does it still solve the way we teach it.' _(ties to: The Yard (scenario correctness))_
7. **Make every refused Yard move name its rule CATEGORY in the feedback (it mostly does — gate it)** — `refuse()` messages already cite categories (lining, securement, kicking, lead foul). Add a playtest assertion that NO refusal is generic ('illegal move') — every refusal must explain WHY against a rule category so a mistake is caught, explained, and remediated in one breath. A silent or vague refusal is misleading feedback, which trains the wrong instinct in a safety tool. _(ties to: The Yard (safety/error review))_
8. **Hard-pin `path.testReady = false` and define the conservative bar in DECISIONS** — Exam is deferred and no 'test-ready' claim may ship. Until an SME sets a breadth threshold (every live domain above a fam floor) AND a retention threshold (mastery survives ≥1 spacing interval, not same-session) AND the exam pool has zero needs-review AND sign-off is recorded, `testReady` must never flip true. Encode this as a guard + a content-check assertion so an optimistic future change can't accidentally claim a trainee is qualification-ready. _(ties to: Exam / path (F3))_

## Failure cases

- **A hardcoded `par` is wrong (e.g. the 7-move puzzle is actually solvable in 6). The game praises the trainee for 'Par!' at a sub-optimal line, teaching an inefficient or rule-bending habit as best practice.** → `tools/solve.mjs` asserts `solve().par === puzzle.par` for all 19 puzzles and fails the build on mismatch. Never edit the solver to agree with a par; fix the par to the proven optimum.
- **A puzzle is unsolvable within the engine's real rules (a layout that can't clear the lead, or a goal car missing from `start`) — the trainee softlocks and blames themselves, exactly the anxious learner this app exists for.** → Harness asserts `solve()` is non-null and that every goal/start car referenced exists. Treat a solver `null` as a hard failure to investigate (raise caps, then redesign), never a pass.
- **A `needs-review` item leaks into a graded Drill/Exam — the cardinal sin F1: a wrong rule taught as correct in a safety trainer.** → Runtime `drillable` already filters trust==='verified'; ADD build-time `content-check.mjs` asserting no non-verified item is reachable by gradable/drillable AND that 'trust' has no typo'd value that would silently mis-classify.
- **A `trust` typo ('verfied') silently drops GOOD content from drills, OR a genuinely-wrong item is mis-flagged 'verified' and sails into grading. The runtime filter obeys the string; it can't know the data is wrong.** → content-check.mjs enums trust to the three legal values and fails on anything else; trust→verified TRANSITIONS require SME sign-off recorded in DECISIONS, never an inline edit.
- **A signal Drill question is ambiguous — `buildQ` asserts correct = item.title, but the randomly chosen aspect is shared by two signal names, so the 'correct' answer is genuinely contestable. F1 on supposedly-verified data.** → Per-aspect diagnosticity check: assert no aspect encoding maps to more than one signal title; SME reviews any collision. Until clean, exclude that aspect from the drill pool.
- **A distractor in a definition question is ALSO true for the stem (two near-synonym definitions), so the trainee who understands the concept gets marked wrong — destroying trust and teaching them to distrust their own correct knowledge.** → Distractor-collision static check (no distractor equals/contains/is-contained-by the answer); flag near-synonyms for SME; never auto-generate distractors from a pool without this guard.
- **SR math drift (code +0.05/ceiling 2.8 vs spec +0.1/no ceiling) makes 'mastered' arrive too fast or too slow — F2 false-mastered: a trainee shown 'qualification-ready' who hasn't actually retained.** → Golden-vector unit test pins the exact interval/ease/fam progression; reconcile spec and code so the number the user sees means what the docs say it means.
- **The Yard refuses a legal-looking move with a vague message, or accepts an illegal one, so the trainee learns the wrong rule. Misleading feedback in a safety context is worse than no feedback.** → Playtest gate: every refusal must name a rule category and be true; every accepted move must be rule-legal per model.js. Cross-check refusal messages against the solver's legal-move set on the optimal line.
- **Someone 'fixes' a failing par by widening solver caps or editing the cost model until the number matches — hiding a real engine bug.** → Revision-loop discipline encoded in HANDOFF: failure triage is data→par→engine, in that order; any change to model.js cost logic re-runs ALL 19 pars, and the diff must be reviewed, not auto-accepted.
- **A future agent flips `path.testReady` true (or the UI implies 'test-ready') before the SME bar is met, making the unsafe claim the app explicitly forbids in V1.** → Hard-pin testReady=false with a guard + content-check assertion; the only way to enable it is an SME sign-off line in DECISIONS plus passing breadth/retention thresholds.

## Validation steps

- `node tools/solve.mjs` exits 0: for all 19 puzzles, solver finds a solution, `solve().par === puzzle.par`, and `replay(solve().opt)` wins at that exact move count. Any mismatch fails the gate.
- `node tools/content-check.mjs` exits 0: zero items with an illegal/typo'd trust value; zero needs-review reachable by gradable/drillable; every drillable rule has ≥1 question; every question.ruleId resolves to a real rule id; every question.answer ∈ its own choices; no duplicate item or question ids.
- `tsc -b --noEmit` passes (schema/types compile-clean) and `oxlint` passes — both folded into `npm run check`.
- SR golden-vector test passes: a fixed answer sequence reproduces the documented interval (0→1→6→round(i*ease)), ease nudge + floor 1.3 + (resolved) ceiling, miss resets interval, fam=min(1,interval/30), and 60-day-half-life decay — matching whichever spec/code is made canonical.
- Distractor-collision check passes: no distractor equals, contains, or is contained by its correct answer; flagged near-synonyms have an SME-review note; signal aspect→title mapping is one-to-one (no aspect diagnostic of two signals).
- Accessibility acceptance passes on a representative Drill + Yard screen: all interactive targets ≥44px; `prefers-reduced-motion` removes Yard animation; read-aloud fires on each new prompt and is silent when offline; focus lands on the Next/feedback control after answering; feedback region is aria-live (confirmed polite).
- Playtest gate (human, one tester per release): each tier-representative puzzle (0,1,2,4,5,6) is winnable to par following in-game hints, has no unrecoverable softlock, and every refused move shows a rule-category-grounded reason. Tester confirms the slice is both useful (teaches the micro-skill) and fun (not frustrating).
- `path.testReady` is verifiably false in a fresh profile and cannot be flipped true without the SME sign-off + threshold guard; the UI nowhere asserts 'test-ready' or 'qualification-ready' as a current capability.
- Determinism check: no graded Yard puzzle is procedurally generated at runtime; the solver is used only to verify authored puzzles (confirmed by code path, not just comment).
- Deploy is blocked on red: a deliberately broken par and a deliberately mis-flagged item each cause `.github/workflows/deploy.yml` to fail before publish (negative test of the gate itself).
